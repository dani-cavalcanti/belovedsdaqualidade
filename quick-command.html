Como criar Quick Commands IDE

Documentação contendo o passo a passo para criação de um Quick Command com base na documentação oficial da ©StackSpot.


Criando o seu Quick Command

Siga os passos para criar:

Passo 1. Acesse o Portal da StackSpot AI

Passo 2. No menu principal, clique em ‘Quick Command’;

Passo 3. Clique no botão ‘Criar Quick Command’;

Passo 4. Selecione a opção: IDE;

Passo 5. Preencha os campos para adicionar informações sobre o seu Quick Command;

  • Nome do Quick Command: adicione um nome claro e descritivo para o seu comando. Exemplo: Technical Debt

  • Comando: adicione um nome que será usado posteriormente. Ele deve seguir o padrão de slug. Para mais informações sobre Slugs, confira o padrão do slug. Exemplo: technical_debt_1

  • Descrição: adicione uma descrição clara e objetiva sobre o que o seu comando faz. Exemplo: Criar uma issue no GitHub destacando a dívida técnica no código selecionado e sugerir uma forma de correção.

Passo 6. Clique no botão 'Próximo'. Você pode escolher entre usar os templates disponíveis (pré-preenchidos) ou criar um template do zero, conforme sua necessidade.

Passo 7. Clique no botão 'Salvar' e confira o resumo do seu Quick Command

1 - Cria documentação de qualidade com base nos requisitos

Nome do Quick Command: cria-doc-qualidade-via-requisito

descrição: Este Quick Command tem por objetivo criar: user story, critérios de aceite, plano de testes e cenários de testes com base na documentação de requisito fornecida.

cria-us-prompt

Sua tarefa é criar a história de usuário, se baseando nos requisitos funcionais do projeto.
Instruções: 
- Forneça a resposta no formato texto sempre em português brasileiro usando as Keywords:
  Quem? Para quem estamos desenvolvendo a funcionalidade.
  O que? Uma descrição resumida da funcionalidade em si.
  Por que? O motivo pelo qual o cliente precisa desta funcionalidade.

  Se possível citando o valor de negócio obtido,
- Revisar a história usando o conceito INVEST:
    Independent (deve ser independente)
    Negotiable (deve ser negociável)
    Valuable (deve agregar valor para o cliente)
    Estimable (deve ser possível estima-la)
    Small (deve ser pequena)
    Testable (deve ser testável)

cria-criterios-de-aceite


Escreva os critérios de aceite seguindo as seguintes regras:
Clareza e Precisão
    Use linguagem clara e específica.
    Evite ambiguidades e termos vagos.
Mensurabilidade
    Os critérios devem ser mensuráveis e verificáveis.
    Inclua métricas ou condições específicas que possam ser testadas.
Relevância
    Os critérios devem estar diretamente relacionados ao requisito.
    Evite incluir informações irrelevantes ou desnecessárias.
Consistência
    Mantenha a consistência com outros requisitos e critérios de aceite.
    Certifique-se de que não haja conflitos entre os critérios.
 Completude
     Os critérios devem cobrir todos os aspectos importantes do requisito.
     Garanta que todas as condições necessárias para a aceitação estejam incluídas.
Testabilidade
     Os critérios devem ser escritos de forma que possam ser testados.
     Inclua cenários de teste claros e específicos.
Acessibilidade
     Os critérios devem ser compreensíveis para todas as partes interessadas.
     Use uma linguagem que seja acessível tanto para desenvolvedores quanto para stakeholders não técnicos.
Formato Padrão
     Utilize um formato padrão para a escrita dos critérios, como o formato Given-When-Then:
        Dado: a condição inicial.
        Quando: a ação que desencadeia o comportamento.
        Então: o resultado esperado.
Validação e Aprovação
     Os critérios devem ser validados e aprovados pelas partes interessadas.
     Inclua um processo de revisão para garantir a aceitação dos critérios.
Documentação
     Documente os critérios de aceite de forma adequada.
     Mantenha um registro atualizado e acessível para todos os membros da equipe.

cria-teste-unitario

Crie testes unitários baseado no  padrão AAA(Arrange, Act, Assert) seguindo as seguintes regras:
            
1. Isolamento: Cada teste unitário deve isolar uma única unidade de código, como uma função ou método, para garantir que o teste seja específico e preciso.
2. Independência: Os testes unitários devem ser independentes uns dos outros. Isso significa que a execução de um teste não deve depender do resultado de outro.
3. Automatização: Sempre que possível, automatize os testes unitários para aumentar a eficiência e garantir a repetibilidade.
4. Cobertura de Código: Os testes unitários devem cobrir todas as partes críticas do código. Utilize ferramentas de cobertura de código para identificar áreas não testadas.
5. Nomenclatura Clara: Nomeie os testes de forma clara e descritiva para que outros desenvolvedores possam entender facilmente o que está sendo testado.
6. Configuração e Limpeza: Cada teste deve configurar seu próprio ambiente e limpar qualquer estado alterado após a execução. Isso garante que os testes não interfiram uns nos outros.
7. Testes de Limite e Equivalência: Utilize técnicas como análise de valor limite e partição de equivalência para garantir que os testes cubram uma ampla gama de entradas possíveis.
8. Mocking e Stubbing: Utilize mocks e stubs para simular dependências externas e garantir que os testes sejam focados na unidade de código específica.
9. Execução Rápida: Os testes unitários devem ser rápidos para executar, permitindo que sejam executados frequentemente durante o desenvolvimento.
10. Documentação: Documente os testes unitários para que outros desenvolvedores possam entender o propósito e a lógica por trás de cada teste.

cria-teste-e2e

 Escreva testes E2E baseado na linguagem da classe e seguindo os seguintes critérios:
 Cobertura Completa do Fluxo de Trabalho:
    Os testes E2E devem cobrir fluxos de trabalho completos do usuário, desde o início até o fim. Isso inclui todas as etapas que um usuário real executaria para concluir uma tarefa.
 Ambiente Realista:
    Os testes devem ser executados em um ambiente que imite o mais próximo possível o ambiente de produção. Isso inclui usar dados reais ou quase reais e configurar o ambiente de teste para refletir a configuração de produção.
 Automatização:
    Sempre que possível, os testes E2E devem ser automatizados para garantir que possam ser executados de forma consistente e repetida sem intervenção manual.
 Validação de Resultados:
    Os testes devem validar não apenas que as ações foram concluídas com sucesso, mas também que os resultados são os esperados. Isso pode incluir verificar o estado do sistema, a saída de dados e a interface do usuário.
  Resiliência e Robustez:
    Os testes devem ser resilientes a pequenas mudanças na interface do usuário e robustos o suficiente para lidar com variações nos dados de entrada.
 Desempenho e Tempo de Execução:
    Embora os testes E2E possam ser mais lentos do que outros tipos de testes, é importante garantir que eles sejam otimizados para desempenho e não demorem excessivamente para serem executados.
 Isolamento:
    Cada teste E2E deve ser isolado e independente dos outros testes. Isso significa que a execução de um teste não deve afetar o estado do sistema de forma que impacte outros testes.
 Manutenção e Clareza: 
    Os testes devem ser escritos de forma clara e fácil de entender, para que possam ser mantidos e atualizados conforme o sistema evolui.
 Integração Contínua: 
    Os testes E2E devem ser integrados ao pipeline de integração contínua (CI) para garantir que sejam executados automaticamente em cada mudança de código.
 Relatórios e Logs: 
    Os testes devem gerar relatórios detalhados e logs que possam ser usados para diagnosticar falhas e entender o comportamento do sistema durante a execução dos testes.

cria-doc-qualidade

 Crie a documentação de qualidade dos testes criados com base nesta lista combina as melhores práticas do Syllabus CTFL 4.0, ISO 29119 (Software Testing) e PMBOK (Project Management Body of Knowledge) para criar uma documentação de qualidade e testes robusta:
1. Planejamento e Estratégia:
    Defina o escopo: Identifique claramente os objetivos, requisitos e recursos da documentação.
    Estabeleça um plano de teste: Defina os tipos de testes, níveis de teste, critérios de entrada e saída.
    Gerencie os riscos: Identifique e analise os riscos relacionados à qualidade da documentação e dos testes.
    Comunique o plano: Compartilhe o plano com as partes interessadas, garantindo clareza e consenso.
2. Documentação de Requisitos:
    Requisitos de teste: Documente os requisitos funcionais e não funcionais a serem testados.
    Casos de teste: Crie casos de teste detalhados, incluindo passos, dados de entrada, resultados esperados e critérios de aprovação.
    Cenários de teste: Descreva situações específicas, como cenários de erro ou de uso crítico, a serem testadas.
    Documentação de requisitos do usuário: Documente os requisitos do usuário final, como interface, funcionalidade e usabilidade.
3. Execução e Relato de Testes:
    Gerenciamento de defeitos: Documente os defeitos encontrados durante os testes, incluindo descrição, passos para reprodução e gravidade.
    Relatórios de teste: Gere relatórios completos e concisos, incluindo resumo dos testes, resultados, defeitos encontrados e recomendações.
    Evidências de teste: Capture e armazene evidências de teste, como logs, screenshots e vídeos, para auditoria e revisão.
    Rastreamento de defeitos: Utilize ferramentas de gerenciamento de defeitos para acompanhar o ciclo de vida dos defeitos, desde a identificação até a resolução.
4. Boas Práticas de Documentação:
    Clareza e concisão: Utilize linguagem clara, concisa e técnica.
    Organização e estrutura: Apresente a informação de forma organizada e estruturada, utilizando tabelas, gráficos e títulos.
    Padronização: Utilize templates e padrões de documentação para manter a consistência.
    Revisão e aprovação: Revise e revise a documentação para garantir a precisão e a qualidade.
    Atualização e controle de versões: Mantenha a documentação atualizada e controle as versões.
5. Regras Específicas:
    CTFL 4.0: Siga as diretrizes e os padrões estabelecidos no Syllabus CTFL 4.0 para documentação e testes.
    ISO 29119: Adote as melhores práticas de testes de software, como testes de integração, sistema e aceitação, descritas na ISO 29119.
    PMBOK: Utilize as ferramentas e técnicas de gerenciamento de projetos do PMBOK para gerenciar a documentação e os testes.
6. Ferramentas e Recursos:
    Ferramentas de teste: Utilize ferramentas de teste automatizado e de gerenciamento de defeitos.
    Documentação online: Utilize plataformas online para armazenamento, compartilhamento e controle de versão da documentação.
    Comunicação eficaz: Utilize ferramentas de comunicação para facilitar a colaboração e o feedback.
